# Contributing to the Ignite Program Repository

Thank you for being part of the Ignite Program! This guide outlines the best practices for contributing to this repository to ensure smooth collaboration across all 8 groups.

## 🌿 Branching Strategy

All development work should be done on dedicated branches. Please follow this naming convention:

```
group{number}/{feature-or-task-name}
```

**Examples:**
- `group1/landing-page`
- `group3/database-schema`
- `group7/api-integration`

The `main` branch is protected and should only receive changes via reviewed Pull Requests.

## 📝 Commit Message Guidelines

Write clear, concise commit messages that describe **what** was changed and **why**.

```
[Group X] Short description of the change

Optional longer description if needed.
```

**Examples:**
- `[Group 2] Add user authentication module`
- `[Group 5] Fix bug in data parsing function`

## 🔀 Pull Request Process

1. Ensure your branch is up to date with `main` before submitting a PR.
2. Fill in the PR template with a clear description of your changes.
3. Request a review from at least one other group member.
4. Address all review comments before merging.
5. Squash commits if necessary to keep history clean.

## 🐛 Reporting Issues

If you encounter a bug or have a feature request, open a GitHub Issue with the following information:

- **Title**: A short, descriptive title.
- **Description**: A detailed description of the issue or feature.
- **Steps to Reproduce** (for bugs): Step-by-step instructions to reproduce the problem.
- **Expected vs. Actual Behavior**: What you expected to happen and what actually happened.

## ✅ Code of Conduct

All participants are expected to treat each other with respect and professionalism. Harassment, discrimination, or disruptive behavior of any kind will not be tolerated.
